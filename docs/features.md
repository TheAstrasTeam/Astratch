# Astratch

> 搭建起玩具和工具的桥梁。

`Astratch` 是一个图形化IDE（集成开发环境），它希望可以让你以“搭积木”的方式搭出*任何东西*，就像Scratch一样。

# 为什么是 Scratch？

> 下文中的 `Scratch` 均指 `Scratch3.0`。对于其它的版本本文不做赘述。

`Scratch`，为麻省理工学院（简写为MIT）媒体实验室下的“终身幼儿园团队”开发的*少儿*编程工具。它被设计的尽可能**简单**来满足给*少儿*用的目标，这也导致它在很多情况下没法满足开发者的需要，例如：

- 你无法在自制积木中使用*返回值*
- 你无法在多个角色之间共享自制积木
- 你无法在广播中附带数据
- 你甚至无法显示一段文字（除非使用 `Scratch Lab` 的艺术字扩展）

这也导致`Scratch`自身的局限性非常高，尽管后来也已有成熟的改版面世，但它们都没有从根本上解决`Scratch`面向少儿所带来无法避免的限制。

并且，`Scratch`已经过于老了，从2019年发布后（实际开发会更早）到现在，已经出现了很多“技术债”，非常不利于维护和扩展。

# Astratch

`Astratch`以`Scratch`为灵感，引入了更为高级的特性，并从0开始开发，使用现代化的技术栈来消除绝大部分“技术债”，这也使得`Astratch`可以做到其它`Scratch`改版很难做到的事——大刀阔斧，修改底层架构。

`Astratch`重新设计了“角色”和“舞台”，将它们替换成了“实体”和“模块”，这和大部分编程语言、游戏引擎一样，但`Astratch`仍确保了它们的简单，来实现“尽管更高级，但也不难理解”的目标。

相较于`Scratch`，**“实体”** 更接近于 **“角色”**，但职责更集中于 **渲染** 而非包揽逻辑、计算等其它工作。

接下来，此文将展开关于`Astratch`新机制的讲解。

## 数据

`Astratch`中的数据功能，大体类似于`Scratch` 的**变量（Variable）**与**列表（List）**，但进行了*大刀阔斧*的修改。

本章将会具体讲解关于此的区别。

### 变量（Variable）和常量 （Constant）

`Astratch` 降低了**列表**的优先级，并且加入了*被广泛使用*的**常量**。变量（Variable）和常量 （Constant）统称为**数据**。在`Astratch`中，**列表**就是一个*较为特殊*的变量。

> 相较于`Scratch`，`Astratch`会通过一个特殊的键值来判断**数据**的`类型（Type）`。

#### 类型（Type）

> 以下的数据不会实际显示为这种文本，它们实际上是通过拖积木构建。所以这里仅供参考

`Astratch`引入了`类型系统`，这不仅是加入了新功能，这还能在*根本上*助力你的开发和项目的实际运行速度，让你的项目**更方便**维护也能让别的`Astratcher`知道你在做什么。

这个功能是**可选的**，如果你没有启用`严格类型`功能，那么`Astratch`会尝试通过上下文**分析**这个数据的类型，如果这个数据被赋予了对应不同的类型的数据，则它会被赋予`any`类型。

> 例如，你将数据`address`定义为了`Sea`，如果后面没再修改数据，那么这个数据将会自动变为`string`类型，如果你在后来又将它改为了`134`，它就会变为`any`。

> ## 你知道吗？
> 
> 实际的数据输入框是*恒为字符串*的，所以这个即便你输入了`123`，它在数据层面仍算作`string`，但你放心，`Astratch`会自动检查这个它能不能是`number`。

 `Astratch`只会自动检查它能不能是`number`（即便你输入的是`true`，它也会是一个`字符串`而不是一个`布尔值`），但对于更多类型，则需通过一个`返回积木`来替换掉这个输入框。这会在下文提供对应的积木说明。

`Astratch`自带以下类型：
- `number` —— 数字（`1`,`25.15`,`-10`）
- `string` —— 字符串 (`Hello World`, `word`, `sky`)
- `boolean` —— 布尔值 （`true`, `false`）
- `array<T>` —— 数组（`[1,3,4]`, `["Peter","John","Chen"]`）
- `object<T,Y>` —— 对象（`{"name":"tomato","types": "fruit"}`）
- `any` —— 未知类型，它是所有类型的集合

> 其中的`T`和`Y`为**泛型**，它的类型是**不确定的**，例如在`array<T>`中，它可以是`array<string>`，也可以是`array<number>`，甚至可以嵌套：`array<object<number, string>>`

##### 联合类型

例如`{"name":"tomato","types": "fruit"}`，它的对应类型便是`object<string, string>`，不过，有人也认为`tomato`（西红柿）也是一个`vegetable`（蔬菜），如果`types`只是一个字符串，未免就不够用了。所以`Astratch`可以让你使用`联合类型`，让这个`types`可以为*多种类型*。例如：
``` JSON
{"name":"tomato","types": ["fruit", "vegetable"]}
```

## Target

在下文，`Target`这个东西将贯穿始终，因此我们很需要在此刻介绍它。简而言之，无论是`模块`还是`实体`，还是`Scratch`的`舞台`和`角色`，在数据层面，都是一个`Target`。

你可以在`Scratch VM`看见它。例如，你可以在浏览器的`devTools`下的**控制台**输入：

```
vm.runtime.targets
```

并按下`回车`（`Enter`）,控制台就会返回一个*数组*，其中的内容便是作品的`舞台`+所有`角色`。

这在 `Astratch` 基本一致，你也可以通过这个命令访问`模块`和`实体`，只不过，具体的格式会有较大的区别，此处不作赘述。

### 关于 “背景”

`Scratch`原生有`背景`功能，它实际上是一个`target`+一个属性值（`isStage`），在`Astratch`中，`背景`将作为一个`实体`工作。

> 你知道吗？通常在`vm.runtime.targets`中的第一项就是舞台！

### 模块

在`Astratch`中，相较于**实体**，**模块**是一个*不负责渲染、专注于封装、处理、导出*的`Target`。

**模块**主要用于封装逻辑、管理数据、导出接口以及实现全局功能，
可被实体或其他模块引用。

**模块**不会拥有**缩放、位置、方向等**外观属性，尽管如此，你依然可以添加这些积木，它会作用在实际使用的**实体**内。

**模块**无法*自动执行*，其中的脚本只能通过`放置到初始化积木`或`导出为函数`来执行。

#### 可见性（Visibility）

你可以将一个数据、一个函数的作用域设置`私有`或`公共`，让不同的模块、实体使用它

如果你将这个数据、函数设为`私有`，尽管你从别的地方导入了它，那么你也无法使用这个数据、函数。除非你额外有一个`公共`的函数使用了它，你才能通过这个函数间接使用那个`私有`的函数和数据

> 如果这个实体/模块修改了另外实体/模块的一个变量的话，则会修改那个被导入模块的数据，因为调用者从来就没它的所有权，（这是引用，不是持有）

### 实体

在`Astratch`中，实体负责实际渲染画面，它持有完整的外观属性。

实体是 `Astratch` 中可被用户直接观察和交互的对象。它拥有渲染所需的数据，并通过调用模块来获得更复杂的能力。实体自身不可被导入，以避免不同对象之间产生直接依赖。

**实体**可以参与`事件`，所以实体
可以在更多的时候被执行，而不只是像模块一样*被动*。

> 实体当然可以持有自己的数据，但它的数据是**私有的**，不可导出，因为实体自身就无法被导入
